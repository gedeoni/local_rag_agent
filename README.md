# Local Dynamic RAG Reasoning Agent

A highly capable, privacy-focused, and dynamic Retrieval-Augmented Generation (RAG) agent constructed using Python and Streamlit. This application empowers users to chat with their documents (PDFs) and web content securely using local LLMs via Ollama (e.g. DeepSeek R1), with a seamless fallback to Cloud Provider APIs (such as OpenAI) if desired.

## 🚀 Key Features

* **Local-First Processing**: Defaults to utilizing local LLMs through Ollama, ensuring your proprietary documents never leave your machine unless explicitly configured otherwise.
* **Hybrid Retrieval System**: Combines native LanceDB vector filtering, robust local python fallbacks, and a raw-text registry to guarantee that relevant document structure is always found.
* **Semantic Query Optimization**: Leverages DSPy to understand vague user queries (e.g., "Summarize this document") and dynamically re-writes them into highly specific context-aware vector search queries using Chain-of-Thought reasoning.
* **Internet Fallback integration**: Configurable agentic fallback utilizing DuckDuckGo tools (via Agno) when internal knowledge bases fail to satisfy a query.
* **Adaptive Response Parsing**: Automatically identifies and cleanly separates "thinking" execution blocks (`<think>...</think>`) generated by reasoning models like DeepSeek, presenting them elegantly in UI expanders.

## 🏗️ Architecture Overview

The codebase is designed with strict separation of concerns across a clean directory structure in mind:

1. **`rag_agent.py`**: The Streamlit Orchestrator. Handles UI logic, application state, configuring the sidebar, and processing the chat execution loop.
2. **`core/retrieval.py`**: The Database & Vector pipeline. Manages LanceDB instances, parses PDFs/Web URLs via Langchain loaders, embeds documents using Ollama/Cloud embeddings, and exposes the unified robust retrieval pipeline.
3. **`core/agents.py`**: The LLM interaction layer. Defines the DSPy `QueryOptimizer` pipeline, and configures the Agno-based `Agent` implementations for generic chat and tool-based web search.
4. **`utils/text_processing.py`**: Centralizes robust Regex parsing to clean messy tags or reasoning logs from outputs.
5. **`utils/system_checks.py`**: Determines environment capabilities (e.g., discovering running Ollama models).

## 🔍 Deep Dive: Key Functions

### `optimize_search_query` (in `rag_agent.py` & `core/agents.py`)
Vector databases interpret language literally. If a user asks *"What is the main conclusion of the uploaded document?"*, a vector query for "uploaded document" is unlikely to mathematically intersect with the actual underlying thesis of the text. 

To solve this:
1. `optimize_search_query` intercepts the user's raw prompt.
2. If RAG is enabled, it gathers the names of all the `processed_documents` in the session.
3. It passes both the raw query and the document names to a **DSPy `QueryOptimizer` module** (located in `core/agents.py`).
4. DSPy utilizes a `ChainOfThought` pipeline guided by a strict `QuerySignature`. The LLM reasons about the prompt and actively replaces vague terms with the specific real-world subjects of your uploaded contexts.
5. The function returns a highly optimized query string, substantially improving LanceDB relevance scoring.

### `retrieve_agent_context` (in `rag_agent.py`)
This function dictates the multi-tiered context discovery pipeline for the final agent. 

1. **Vector Retrieval**: It first invokes `should_use_vector_search()`. If valid, it dives into `execute_retrieval_pipeline` (in `core/retrieval.py`).
   * *Phase A*: It attempts a strict metadata-filtered LanceDB retrieval (searching only within the documents the user manually selected in the sidebar). 
   * *Phase B*: If the native LanceDB engine fails to parse the filter correctly, it catches the error and performs a broad similarity search, strictly validating the source documents using Python.
   * *Phase C*: If similarity scores are lower than the threshold limit, it performs a raw database lookup in a dedicated `document_registry` table, feeding the raw full-text to the context window as an absolute last resort.
2. **Web Fallback**: It utilizes `should_use_web_fallback()` to check if the Vector layer came up entirely empty (or if the user toggled the Force Web Search button). If invoked, it constructs an Agno `Web Search Agent` equipped dynamically with DuckDuckGo querying tools to summarize live information.
3. **Return**: It seamlessly hands the unified `context` string and corresponding `docs` array back to `rag_agent.py` for text-generation.
